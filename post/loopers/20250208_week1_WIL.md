# Week 1 WIL — TDD로 시작한 첫 부트캠프, 그리고 결정들

---

## 서론

이번 주는 나에게 두 가지 처음이 시작된 주였다. 첫 번째는 **부트캠프에 참여한 것**, 두 번째는 **실제 프로젝트에서 TDD를 적용한 것**이다.

요구사항은 단순해 보였다. 유저 로그인과 회원가입 기능을 구현하는 것. 필요한 정보는 { 로그인 ID, 비밀번호, 이름, 생년월일, 이메일 }이고, 비밀번호 암호화, 유효성 검증, 내 정보 조회, 비밀번호 변경 등 전형적인 사용자 인증 플로우였다.

하지만 막상 시작하니 예상보다 많은 고민이 필요했다. CS를 전공했지만, 실제로 코드를 작성하려니 "어떤 아키텍처를 선택할 것인가", "클래스 간 역할을 어떻게 나눌 것인가", "마감일 안에 완성할 수 있는 현실적인 선택은 무엇인가"라는 질문들이 쏟아졌다.

그리고 TDD. 테스트를 먼저 작성하고 구현을 나중에 한다는 개념은 알고 있었지만, 실제로 해보니 생각보다 어려웠고, 동시에 명확했다.

---

## 본론

### TDD, 머리로는 알았지만 손으로는 낯설었다

TDD의 기본 사이클은 간단하다. **Red → Green → Refactor**. 실패하는 테스트를 작성하고, 통과시키고, 리팩토링한다.

하지만 실제로 해보니 처음엔 막막했다. "테스트를 먼저 작성한다"는 것이 구체적으로 무엇을 의미하는지, 어디서부터 시작해야 하는지 감이 오지 않았다.

```kotlin
// 예를 들어, 이런 테스트부터 작성했다
@Test
fun `이미 가입된 로그인 ID로는 가입이 불가능함`() {
    // Given
    val existingUserId = "duplicateUser"
    whenever(userRepository.existsByUserId(existingUserId)).thenReturn(true)

    // When & Then
    assertThrows<ConflictException> {
        userService.createUser(
            userId = existingUserId,
            password = "Password123!",
            name = "홍길동",
            birthDate = LocalDate.of(1990, 1, 1),
            email = "test@example.com"
        )
    }
}
```

테스트를 작성하면서 자연스럽게 "어떤 예외를 던져야 하는가", "어떤 파라미터가 필요한가", "Repository는 어떤 메서드를 제공해야 하는가"가 명확해졌다. **테스트가 설계를 유도하는 경험**이었다.

### Test Doubles, 이름만 들었던 개념들

TDD를 하면서 자연스럽게 마주한 것이 **Test Doubles**였다. Mock, Stub, Spy, Fake... 용어는 들어봤지만 실제로 언제, 어떻게 써야 하는지는 막연했다.

이번 주에 ["테스트, 묻고 doubles로 가"](../20250206_테스트_묻고_doubles_로_가.md)라는 글을 작성하면서 각각의 역할을 정리했다:

- **Stub**: 정해진 답변만 하는 바보 객체. 테스트에 필요한 최소한의 응답만 제공.
- **Mock**: 행동까지 감시하는 객체. 특정 메서드가 호출되었는지 검증.
- **Spy**: 실제 객체를 래핑해서 감시. 대부분은 실제 구현을 사용하되 필요한 부분만 stub 가능.
- **Fake**: 실제처럼 동작하는 단순 구현. 예: H2 in-memory DB.

멘토님들은 **실전에서는 Mock과 Stub을 주로 사용한다**고 조언해 주셨다. 그리고 때로는 E2E 테스트가 더 나은 선택일 수 있다고도 했다. 단위 테스트로 모든 걸 검증하려다 보면 오히려 테스트가 구현에 강하게 결합되어 리팩토링이 어려워지는 "Mock Hell"에 빠질 수 있다는 것이다.

### 코드 아키텍처, 어떤 길을 선택할 것인가

이번 주 가장 큰 고민은 **코드 아키텍처를 어떻게 설계할 것인가**였다.

크게 두 가지 선택지가 있었다:
- **A**: `Controller → Facade → Service` 계층 구조로 클린 아키텍처를 지향
- **B**: `Controller → Service` 직접 호출로 단순하게 시작

결국 **B를 선택했다**. 이유는 명확했다:
1. 현재 요구사항의 비즈니스 로직이 단순하다
2. 단일 도메인(User)에 집중되어 있다
3. 불필요한 복잡도를 줄이고 빠르게 검증하고 싶었다

하지만 이는 **트레이드오프**였다. 초기 생산성을 높였지만, 추후 비즈니스 로직이 복잡해지면 리팩토링(Facade 도입, 모델 분리)이 필요할 것이다.

또 다른 결정은 **JPA Entity = Domain Model**로 통합한 것이었다. `UserModel`이 영속성 객체이면서 동시에 도메인 모델 역할을 했다. 이 역시 초기 개발 속도를 위한 선택이었지만, 도메인 로직이 복잡해지면 **Domain Entity와 JPA Entity를 분리**해야 할 것 같다는 생각이 들었다.

```kotlin
// 현재: JPA Entity = Domain Model
@Entity
@Table(name = "users")
class UserModel(
    @Column(nullable = false, unique = true)
    val userId: String,

    @Column(nullable = false)
    var encryptedPassword: String,

    // ... 도메인 로직과 영속성이 한 곳에
) : BaseEntity() {
    fun updatePassword(newEncryptedPassword: String) {
        this.encryptedPassword = newEncryptedPassword
    }
}
```

멘토님들의 세션과 다른 사람들의 PR을 구경하며, 내가 구현한 구조의 의도인 **레거시 코드 스타일과 클린 아키텍처의 중간 지점** 를 시작으로, 내가 점진적으로 적용할 수 있는 개선/리팩토링을 그릴 수 있어야 하는 트레이드오프가 존재한다는 점.

이를 위해 필요한 설계도의 의도가 결국 테스트코드이며, TDD는 이 의도를 먼저 작성하고, 실제 코드를 작성한다는 점을 경험해보았다.

### 인증, Spring Security 없이 시작하기

Week 1 요구사항은 복잡한 인증/인가 시스템이 아니었다. 그래서 Spring Security Filter 대신 **커스텀 헤더**(`X-Loopers-LoginId`, `X-Loopers-LoginPw`)를 사용해 명시적으로 인증을 구현했다.

```kotlin
@GetMapping("/me")
fun getMyInfo(
    @RequestHeader(AuthHeader.HEADER_LOGIN_ID) loginId: String,
    @RequestHeader(AuthHeader.HEADER_LOGIN_PW) loginPw: String
): UserResponse {
    val user = userService.authenticate(loginId, loginPw)
    return UserResponse.fromMasked(user)
}
```

이 역시 **트레이드오프**였다. 초기에는 간단하지만, 추후 인증/인가 로직을 Interceptor나 Spring Security Filter로 이관해야 할 것이다.

### 테스트 커버리지, 얼마나 작성해야 하나

TDD로 진행하다 보니 자연스럽게 다양한 레벨의 테스트가 작성되었다:

- **단위 테스트** (`UserServiceUnitTest`): MockK를 사용해 Service 로직 검증
- **통합 테스트** (`UserServiceTest`, `UserRepositoryTest`): SpyBean으로 실제 DB 연동 검증
- **E2E 테스트** (`UserV1ApiE2ETest`): HTTP 요청부터 응답까지 전체 플로우 검증

특히 비밀번호 정책(8~16자, 영문 대소문자/숫자/특수문자, 생년월일 포함 불가)은 단위 테스트로 세세하게 검증했고, 전체 플로우는 E2E 테스트로 보장했다.

멘토님들은 "E2E 테스트는 최소한으로 유지하되, 핵심 플로우는 반드시 포함하라"고 조언했다. E2E 테스트는 느리고 깨지기 쉽지만, 전체 시스템이 실제로 동작하는지 확인하는 가장 확실한 방법이기 때문이다.

## 결론

첫 주를 마무리하며 느낀 점들을 정리하면:

### 배운 것들

1. **TDD는 설계 도구다**: 테스트를 먼저 작성하면 자연스럽게 인터페이스가 명확해지고, 의존성이 분리된다.
2. **Test Doubles의 실전 활용**: Mock과 Stub의 차이를 이해했고, 언제 무엇을 써야 하는지 감을 잡았다.
3. **아키텍처는 트레이드오프다**: "완벽한 설계"는 없다. 현재 요구사항과 마감일을 고려해 현실적인 선택을 해야 한다.
4. **레거시와 클린 아키텍처의 중간**: 모든 걸 처음부터 클린하게 설계할 필요는 없다. 단순하게 시작하고, 복잡도가 올라가면 리팩토링하는 것도 전략이다.

### 여전히 고민 중인 것들

1. **Domain Entity와 JPA Entity 분리**: 지금은 통합했지만, 도메인 로직이 복잡해지면 분리가 필요할 것 같다. 다음 주에 시도해 볼 예정이다.
2. **인증/인가 전략**: 커스텀 헤더로 시작했지만, 언제 Spring Security나 JWT로 넘어가야 할까?
3. **Facade 패턴 도입 시점**: 언제 Controller → Service에서 Controller → Facade → Service로 확장해야 할까?

### 다음 주 계획

다음 주에는 팀원들과 함께 **PRD(요구사항 정의서), 다이어그램, ERD**를 작성하기 전에 다음을 시도해 볼 것이다:

1. **Domain Entity와 JPA Entity 분리**: 도메인 로직을 영속성 관심사에서 분리해 보기
2. **인증 어노테이션 적용**: 커스텀 헤더 대신 `@Authenticated` 같은 어노테이션으로 인증 로직 분리하기

첫 주였지만 많은 것을 배웠고, 많은 결정을 내렸다. 완벽하지 않았지만, **움직이는 코드를 만들었고, 테스트로 검증했으며, 왜 이렇게 설계했는지 설명할 수 있다**. 그것만으로도 충분한 한 주였다.

TDD는 어려웠지만, 다시 돌아가도 같은 선택을 할 것 같다. 테스트가 주는 안정감과 설계의 명확함은 초기의 어색함을 충분히 보상했다 생각한다.

코딩에는 정답은 없다. 하지만, 내가 작성을 했든, Agent 가 작성을 했든, 그 구조와 의도는 내가 알고 있으며, 언제든 수정할 수 있어야하는 주체가 되어야 한다.

요즘 항상 느끼던 바를 실습과 멘토링으로 다시한번 상기시킬 수 있는 좋은 기회였다.
